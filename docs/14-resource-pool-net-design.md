## Thesis: resources are places, claims are tokens

A pool of N units of some scarce infrastructure (GPUs, LLM inference slots, lab
machines, human workers) is modelled as a **place holding N capacity tokens**.
Acquiring the resource consumes a token; releasing returns one. This is the
textbook Petri-net use case, and it gives us — *for free, from the firing rule* —
what DAG schedulers bolt on:

- **Admission control** — a grant transition needs a token in BOTH the claim
  inbox AND the pool; an empty pool simply leaves it disabled, so claims queue.
  (`find_valid_binding` requires a token at every input place.)
- **Mutual exclusion** — at most N capacity tokens exist, so at most N holds are
  ever active. The engine fires one transition per net per step, so grants never
  race (`select_next_transition`).
- **Backpressure & fairness** — queued claims wait in the inbox; the
  earliest-enabling-time tiebreaker serves them roughly FIFO, and a `priority`
  expression gives weighted queues.
- **Auditability & replay** — the marking is a fold over the event log; the whole
  contention history is replayable and time-travelable.

The resource-pool net is the direct generalization of the existing
`scheduler-net` (`engine/sdk/examples/scheduler_net.rs`): instead of relaying a
job to a scheduler backend, it grants and reclaims a fixed pool of capacity.

## Net design (claim / grant / release / reap)

```text
                    resource-pool-net  (long-lived, single owner of the pool)

  claim_inbox  ─┐                                        ┌─▶ grant_outbox ─(reply "grant")─▶ instance/grant_inbox
  (bridge_in)   ├─▶ [t_grant] consume claim + 1 capacity ┘
  pool ─────────┘            (emits ONLY the bridge grant reply)

  release_inbox ─▶ [t_release] consume release request ─▶ pool (+1 capacity) + done
  (bridge_in)

  lease_expired ─▶ [t_reap]   (crash recovery — see "Lease reaping")
  (signal)
```

Instance side (one per workflow instance), generated by the compiler (M3):

```text
  p_in ─[t_prepare: mint grant_id]─▶ claim_out ─(bridge → pool/claim_inbox, reply "grant")
  grant_inbox ─[t_acquired]─▶ body ─┬─ success ─▶ p_output  + release_out
                                    ├─ error   ─▶ p_error   + release_out
                                    └─ cancel  ─▶ p_cancel  + release_out
  release_out ─(bridge → pool/release_inbox)
```

**Every body-exit edge MUST arc to `release_out`** — a forgotten release strands
a capacity token and, under contention, deadlocks the pool. This is enforced as a
compile-time structural invariant (M3) and backstopped by lease reaping.

## The reply-routing taint (load-bearing gotcha — found in M2)

`firing.rs::route_output_tokens` (~line 230) stamps a firing's
`consumed_reply_routing` onto **every internal output token** of a transition
that consumed a routed (bridged-reply) token. Consequence: a naive pool where
`t_grant` consumes the routed claim AND emits an `in_use` hold taints the hold
with the claim's `grant` reply channel; recycling capacity from that hold in
`t_release` produces a capacity token still carrying the *old* claim's routing.
The next `t_grant` then tries to bind `{recycled cap (req-0 routing), new claim
(req-1 routing)}`, `merge_reply_routing` rejects the conflicting `grant` channel,
`find_valid_binding` returns `None`, and — because a guardless transition's fast
path never tries other combinations — **the pool wedges after the first
release**.

### Rules that fall out of this

1. **Keep capacity tokens clean.** The token recycled into the pool must be
   produced by a transition that consumed *no* routed token. In M2 the pool holds
   clean capacity tokens, `t_grant` emits **only** the bridge grant reply (a
   bridge output produces no tainted local token), and `t_release` returns a
   clean capacity token built from the **clean** release request (it crossed a
   plain `bridge_out`, so its inbound injection carries no reply routing).
2. **A transition cannot consume from a `bridge_reply` place.** The grant landing
   place on the instance must be `internal` (bridged tokens land by name
   regardless of kind, but only consumable kinds can be transition inputs).
3. **`correlate(a, b, field)` emits a Rhai guard** `a.field == b.field`.
4. **`evaluate_until_quiescent` fires all enabled transitions**, so gate any
   "finish/release" step behind an explicit signal when you need a stable,
   observable hold.

### Clean `in_use` + lease reaping (for M4/M5)

A counter-pool (capacity tokens only, no `in_use` record) cannot reap a crashed
holder, because nothing records the outstanding hold. To get an observable
`in_use` AND crash recovery without re-introducing the taint, the holder
**registers** its hold back to the pool over a *plain* `bridge_out` after it
receives the grant:

```text
  instance: grant_inbox ─[t_register]─▶ register_out ─(bridge → pool/register_inbox)
  pool: register_inbox ─[t_admit]─▶ in_use  (CLEAN hold {grant_id, gpu_id})
  pool: t_release / t_reap consume the CLEAN in_use hold → clean capacity back
```

Because the registered hold crossed a plain bridge (no reply routing), consuming
it in `t_release`/`t_reap` keeps the recycled capacity clean. Lease reaping is a
`timer_schedule` (`ctx.delay`) armed at admit time that emits `lease_expired`;
`t_reap` joins it with the matching `in_use` hold. The timer keeps reclamation on
the deterministic, journaled effect-replay path (never a wall-clock guard) — which
is exactly the M1 replay-determinism property.

## Implementation pointers

- Pool net (deployable artifact): `engine/sdk/examples/resource_pool_net.rs`
  (`cargo run -p aithericon-sdk --example resource_pool_net -- --deploy --net-id resource-pool-net`).
- M1 proof: `engine/core-engine/crates/test-harness/tests/resource_pool.rs`.
- M2 proof: `engine/core-engine/crates/test-harness/src/integration/resource_pool.rs`.
- M3 lowering (BUILT): `lower_automated_step_pooled` in
  `service/src/compiler/lower/automated_step.rs` (delegated when the new optional
  `resourcePool` field on AutomatedStep — sibling to `deploymentModel` — is
  present; absent ⇒ byte-identical inline path). It wraps the full inline
  executor lifecycle: `t_{id}_claim` (mint `grant_id`, `bridge_out_reply_channels`
  → `resource-pool-net/claim_inbox`, reply `grant`→`p_{id}_grant_inbox`, park
  `p_{id}_pending`); `t_{id}_acquire` (consume pending+grant, build job spec,
  `bridge_out` `register_out`, park `p_{id}_held`); `t_{id}_to_output` and
  `t_{id}_to_error` both consume `p_{id}_held` and `bridge_out` a release. The
  retry-exhausted edge is routed to a dedicated `p_{id}_exhausted` consumed by
  `t_{id}_to_error` so EVERY reachable terminal exit releases exactly once
  (`dead_letter` is an unreachable lifecycle sink). `grant_id =
  input._instance_id + ":<node_id>"` — globally unique + replay-deterministic
  (the launcher stamps `_instance_id`; `YIELD_LOGIC` preserves `_`-prefixed keys
  on every control token). Constants in `service/src/compiler/well_known.rs`:
  `RESOURCE_POOL_NET_ID` + `POOL_{CLAIM,REGISTER,RELEASE}_INBOX`.
- Engine cross-net mechanics: `engine/core-engine/crates/nats/src/cross_net_bridge.rs`,
  `firing.rs::route_output_tokens`, `binding.rs::{find_valid_binding,merge_reply_routing}`.

## Showcase scenario

"GPU rendering pool" — N=2 GPUs, K=4 concurrent render jobs (mock ~6s timer
body). Exactly 2 run while 2 queue. The pool place visibly drains 2→0 and refills
as jobs release; the instance list shows 2 "Running (GPU held)" and 2 "Waiting for
GPU". Money shot: kill a running instance → its lease is reaped → the freed GPU is
immediately granted to a waiting instance.

## Run the showcase (live dogfood)

One-command per the plan, against a `just dev` stack running mekhan built from
this branch (the demo's AutomatedStep declares `resourcePool`, which only the
M3 compiler lowers). The recipes live in `just/dev.just`.

```bash
just dev                       # full stack (engine :13030, mekhan :13100, app :15173)
just dev up-mekhan             # ensure mekhan is the M3 build, if the stack predates it
just dev resource-pool-up      # build + deploy + activate resource-pool-net (N=2)
just dev resource-pool-demo 4  # publish 13-resource-pool + fire 4 instances (start_token job_name)
```

Open `http://localhost:15173/nets/resource-pool`: the pool drains 2→0, two
instances render while two queue in `claim_inbox`, and each release re-grants a
slot. In an instance graph view (`/instances/{id}`) the queued nodes show the
purple **"waiting"** badge (`p_{id}_pending`>0 ∧ `p_{id}_held`==0).

Money shot — crash a holder and watch the lease reaped + re-granted. Take a
running instance's id and inject its lease expiry (the deployed pool net's
`lease_expired` is a signal place; a durable timer feeds it in production):

```bash
engine/target/debug/aithericon inject resource-pool-net lease_expired \
  '{"grant_id":"<instanceId>:render"}'
```

The freed GPU returns to `pool` and is immediately granted to a waiter.

> Why this wasn't auto-run here: the dev engine/mekhan on :13030/:13100 is a
> shared singleton and another session had it up; loading the M3 mekhan needs a
> restart that would disrupt concurrent work. The dynamics are already proven
> headless (M4 over a real NATS testcontainer) and the lowering offline (M3
> `compiler_e2e` + `air_snapshots` golden); this runbook is the only step that
> touches a live shared stack.

## Productionization gate (out of scope for the prototype)

- Registry-resolved per-workspace pools (`resource_alias` → pool net id via
  `service/src/petri/resource_resolver.rs`) instead of the global well-known id.
- Capability sharding (`pool_<kind>` places + per-kind grant transitions) to
  remove the `find_valid_binding` Cartesian cost and add heterogeneous matching.
- Marking snapshots for the immortal pool net (`projection.rs::project_marking` is
  a full fold from event 0; a never-terminating net needs periodic
  `(event_seq, Marking)` snapshots before it accumulates a large log).
